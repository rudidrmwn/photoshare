Photo Sharing Application ER Diagram

This diagram illustrates the MongoDB schema design and relationships between User, Photo, and Comment models.

erDiagram USER { string username string email string password } PHOTO { ObjectId owner string url string thumbUrl number width number height string mime number size string caption string[] tags number likes ObjectId[] likedBy number commentsCount } COMMENT { ObjectId photo ObjectId user string text } USER ||--o{ PHOTO : owns USER ||--o{ COMMENT : writes PHOTO ||--o{ COMMENT : has USER }o--o{ PHOTO : likes
Entity/Collection
Required Field
Indexed Field
Reference Field

User Model

username string (required, unique)
email string (required, unique)
password string (required)

Includes pre-save hook to hash password with bcrypt

Includes method to compare passwords

Photo Model

owner ObjectId (ref: User) indexed
url string (required)
thumbUrl string (required)
tags string[] indexed
likes number indexed
likedBy ObjectId[] (ref: User) indexed
commentsCount number

Includes text indexes on caption and tags

Includes timestamps

Comment Model

photo ObjectId (ref: Photo) indexed
user ObjectId (ref: User)
text string (required)

Includes timestamps

Relationships

User → Photo
One-to-Many
A user can own many photos, but each photo has only one owner
User → Comment
One-to-Many
A user can write many comments, but each comment is written by one user
Photo → Comment
One-to-Many
A photo can have many comments, but each comment belongs to one photo
User ↔ Photo
Many-to-Many
A user can like many photos, and a photo can be liked by many users (via likedBy array)
